## 🚨 當前需要修復的重要問題：重建 `gui.py`

**問題描述**： 
先前產生的程式碼（包含 `guitemp.py` 與 `gui.py`）因為檔案存檔時的字元編碼不一致，導致所有中文標籤、字串都變成了亂碼（Mojibake，例如 `?Şč?č˛Ź...`），同時這些亂碼中含有未閉合的引號，造成 Python 直譯器報出 `SyntaxError: unterminated triple-quoted string literal` 等語法錯誤。

**下一步應對方案：**
不花費時間手動修復亂碼檔案，請 **直接從零重新撰寫 `gui.py`**。
在重新撰寫時：
1. 必須確保檔案開頭宣告 `# -*- coding: utf-8 -*-`。
2. 介面上所有標籤、按鈕、提示文字，請直接使用正常的**繁體中文**。
3. 參考 `system_spec.md` 中定義的參數、資料夾結構（例如 `models.py` 已經有寫好的 DataClass），與下方列出的「後續需求清單」，把 GUI 的殼重新組裝起來。

## 後續需求清單 (本次重建 GUI 需一併實作的功能)

### 1. 送出執行時，需要確認所有 folder 及輸入內容是否完整
- 執行前集中驗證所有必填欄位
- 驗證內容、作者、Query Template、日期區間是否為有效值
- 缺任一必要欄位時不可執行
- 路徑錯誤時要明確指出是哪一個欄位，將錯誤整理成清單一次顯示。

### 2. 驗證 SQL 是否正常，引號是否正確，語法有沒有錯誤
- 增加 SQL 靜態檢查流程。檢查單引號是否成對、檢查括號是否平衡。
- 檢查常見危險格式，例如空白 SQL、只有註解、明顯截斷的語句。
- 將檢查結果顯示在 GUI，區分錯誤與警告。

### 3. 驗證 DELETE 是不是有加 WHERE
- 若有 `DELETE`，驗證該語句是否包含 `WHERE`。
- 若未加 `WHERE`，直接視為阻擋錯誤，不允許執行。 `DELETE FROM table;` 必須被擋下。

### 4. SQL 可以讀取某一個 file 內容，或是可以在畫面直接輸入內容
- 新增 GUI 文字區塊，可直接貼上原始 SQL。
- 增加輸入模式切換下拉選單或 Radio 按鈕 (`從檔案讀取`, `直接輸入 SQL`)。
- 當選擇直接輸入時，SQL 檔案欄位可禁用；當選擇檔案讀取時，載入文件內容進行預覽。

### 5. 畫面上有開始及結束日期，這兩個日期可以帶值到輸出的 SQL 區間
- GUI 新增「開始日期」與「結束日期」欄位 (文字框搭配日期選擇小視窗)。
- 支援 `${startDate}`、`${endDate}` 等日期占位符替換。
- **需要實作日期選擇器邏輯 (簡易 Calendar)** 讓使用者點擊選擇。

### 6. 畫面上可以按不同 tab，有兩個可以觀看，一個是模板輸出後的結果，一個是原始 SQL
- GUI 下方區塊需實作 `ttk.Notebook` 分頁介面。
- 第一個分頁顯示 `原始 SQL`，第二個分頁顯示 `模板輸出結果`。
- 在執行前可進行修改並預覽，執行後顯示最終結果。

### 7. SQL 的內容要有語法亮度
- 為原始 SQL 與輸出結果兩個視圖加入語法高亮 (使用 Tkinter Text 的 tag 功能)。
- 高亮 SQL 的關鍵字 (SELECT, WHERE, DELETE, UPDATE 等)、字串、註解、數字。

## 建議實作順序

此順序為**重建 `gui.py` 的實作步驟建議**，請直接產生一個完整可運作的 `gui.py` 腳本：

1. **第一步（版面配置）**：引入 tkinter、建構視窗、佈局表單元件（輸入欄位、檔案選取按鈕）、放置 Notebook 頁籤與 Text 顯示框。確保繁體中文字串乾淨無亂碼。
2. **第二步（綁定功能）**：將按鈕綁定至開啟檔案對話框(`filedialog`)，綁定日期小視窗，實作按下「產生SQL」後的邏輯。
3. **第三步（實作驗證邏輯）**：在執行按鈕的 callback 中加入對必填文字、是否存在檔案以及 SQL DELETE / 引號的驗證，有錯則彈出 `messagebox.showerror`。
4. **第四步（預覽與高亮）**：把結果顯示在 Notebook 頁籤的 Text 框內，並套用自製基礎關鍵字高亮邏輯（掃描 `SELECT`, `FROM` 等並加亮字體顏色）。
